# 청구항 작성 가이드

## 청구항이란?

청구항(Claims)은 **특허권의 범위를 정의**하는 가장 중요한 부분입니다.
명세서는 발명을 설명하지만, 실제 권리는 청구항에 기재된 내용만 보호됩니다.

---

## 청구항 종류

### 1. 독립항 (Independent Claim)
- 다른 청구항을 인용하지 않음
- 발명의 핵심 구성요소만 포함
- **가장 넓은 권리범위** 정의
- 보통 1~3개

### 2. 종속항 (Dependent Claim)
- 다른 청구항(독립항 또는 다른 종속항)을 인용
- 추가적인 한정 사항 기재
- 독립항이 무효화되어도 종속항은 유지될 수 있음
- 독립항 보호를 위한 안전장치

---

## 청구항 형식

### 장치/시스템 청구항 (Apparatus Claim)
```
[발명의 명칭]에 있어서,
[구성요소 A]와;
[구성요소 B]와;
[구성요소 C]를 포함하는 것을 특징으로 하는 [발명의 명칭].
```

### 방법 청구항 (Method Claim)
```
[발명의 명칭] 방법에 있어서,
[단계 A]하는 단계와;
[단계 B]하는 단계와;
[단계 C]하는 단계를 포함하는 것을 특징으로 하는 [발명의 명칭] 방법.
```

### 컴퓨터 프로그램 청구항
```
컴퓨터로 하여금 [기능]을 수행하게 하는 명령어들을 포함하는
컴퓨터 판독 가능한 기록매체에 있어서,
[단계 A]를 수행하는 명령어와;
[단계 B]를 수행하는 명령어와;
[단계 C]를 수행하는 명령어를 포함하는 것을 특징으로 하는
컴퓨터 판독 가능한 기록매체.
```

## 독립항 작성 전략

### 원칙 1: 최소 필수 구성요소만
```
[나쁜 예 - 너무 한정적]
"빨간색 버튼과 10cm 화면을 가진 스마트폰"
→ 파란색 버튼이면 침해 아님

[좋은 예 - 넓은 범위]
"입력부와 표시부를 포함하는 휴대용 전자장치"
→ 색상, 크기 관계없이 침해 가능
```

### 원칙 2: 상위 개념 사용
```
[나쁜 예] "스마트폰의 터치스크린"
[좋은 예] "사용자 입력을 수신하는 입력부"

[나쁜 예] "MySQL 데이터베이스"
[좋은 예] "데이터를 저장하는 저장부"
```

### 원칙 3: 기능적 표현 활용
```
[나쁜 예] "GPS 센서"
[좋은 예] "사용자의 위치 정보를 획득하는 위치 획득부"
```

## 종속항 작성 전략

### 구조
```
제[N]항에 있어서,
상기 [구성요소]는 [추가 한정사항]인 것을 특징으로 하는 [발명명칭].
```

### 종속항 유형

**1. 구체적 구현 한정**
```
제1항에 있어서,
상기 분석부는 머신러닝 알고리즘을 이용하여 분석하는 것을 특징으로 하는 시스템.
```

**2. 추가 구성요소**
```
제1항에 있어서,
상기 시스템은 사용자 피드백을 수신하는 피드백 수신부를 더 포함하는 것을 특징으로 하는 시스템.
```

**3. 수치 한정**
```
제1항에 있어서,
상기 예측부는 85% 이상의 정확도로 예측하는 것을 특징으로 하는 시스템.
```

**4. 다중 인용 (종속항의 종속항)**
```
제2항에 있어서,
상기 머신러닝 알고리즘은 딥러닝 신경망인 것을 특징으로 하는 시스템.
```

## 청구항 세트 예시

### 예시: AI 기반 실행 계획 시스템

```
【청구항 1】 (독립항 - 장치)
사용자의 행동 패턴 데이터를 수집하는 데이터 수집부와;
상기 행동 패턴 데이터를 분석하여 최적 실행 시점을 예측하는 분석부와;
행동과학 이론에 기반하여 실행 계획을 생성하는 계획 생성부와;
상기 최적 실행 시점에 상기 실행 계획을 사용자에게 제공하는 출력부를
포함하는 것을 특징으로 하는 실행 계획 생성 시스템.

【청구항 2】 (종속항)
제1항에 있어서,
상기 행동과학 이론은 구현 의도(Implementation Intention),
초소형 습관(Tiny Habits), 습관 스태킹(Habit Stacking),
시간적 동기이론(Temporal Motivation Theory) 중
적어도 하나를 포함하는 것을 특징으로 하는 실행 계획 생성 시스템.

【청구항 3】 (종속항)
제1항에 있어서,
상기 분석부는 사용자의 과거 실행 성공률, 시간대, 위치 정보를 기반으로
최적 실행 시점을 예측하는 것을 특징으로 하는 실행 계획 생성 시스템.

【청구항 4】 (종속항)
제1항에 있어서,
사용자의 음성 입력을 인식하여 할 일을 등록하는 음성 인식부를 더 포함하는
것을 특징으로 하는 실행 계획 생성 시스템.

【청구항 5】 (독립항 - 방법)
컴퓨터로 구현되는 실행 계획 생성 방법에 있어서,
(a) 사용자의 행동 패턴 데이터를 수집하는 단계와;
(b) 상기 행동 패턴 데이터를 분석하여 최적 실행 시점을 예측하는 단계와;
(c) 행동과학 이론에 기반하여 실행 계획을 생성하는 단계와;
(d) 상기 최적 실행 시점에 상기 실행 계획을 사용자에게 제공하는 단계를
포함하는 것을 특징으로 하는 실행 계획 생성 방법.

【청구항 6】 (종속항)
제5항에 있어서,
상기 (c) 단계는,
사용자의 할 일을 2분 이내에 완료 가능한 초소형 행동으로 분해하는 단계를
포함하는 것을 특징으로 하는 실행 계획 생성 방법.

【청구항 7】 (독립항 - 기록매체)
제5항 내지 제6항 중 어느 한 항에 따른 방법을 컴퓨터에서 실행시키기 위한
프로그램을 기록한 컴퓨터로 읽을 수 있는 기록매체.
```

## 청구항 작성 체크리스트

- [ ] 독립항이 너무 좁지 않은가? (필수 구성요소만 포함)
- [ ] 종속항으로 실시예를 충분히 커버하는가?
- [ ] 장치 청구항과 방법 청구항 모두 포함했는가?
- [ ] 명세서에서 모든 청구항 내용을 뒷받침하는가?
- [ ] 선행기술과 차별되는 구성요소가 명확한가?
- [ ] 용어가 명세서와 일관되게 사용되었는가?

---

## 주의사항

1. **청구항 = 권리범위**: 청구항에 없으면 보호 안됨
2. **넓게 쓰되 선행기술 회피**: 선행기술과 겹치면 거절
3. **명세서 뒷받침 필수**: 청구항 내용이 명세서에 없으면 거절
4. **명확성**: 모호한 표현 금지 ("적절한", "효과적인" 등)
5. **단일성**: 하나의 발명 개념으로 통일
